## Race Page: Participant Search with Percentiles & Y-o-Y Progress (D1+D2)

**Приоритет:** Low
**Сложность:** Medium
**Добавлено:** 2026-03-02
**Источник:** step 5.5d (D1, D2) — отложено как избыточный UI

### Суть

Добавить на страницу гонки блок "Найди себя" — поиск участника по имени с персентилями и таблицей прогресса по годам. Использует существующий API `GET /races/{raceId}/search?name=`.

### Почему отложено

На странице гонки уже есть: глобальный поиск в navbar + фильтр в таблице результатов. Третий поиск — перебор. Возможно стоит интегрировать персентили и прогресс в существующий UI, а не создавать отдельный блок.

### Что было запланировано

- **D1**: Компонент `SearchResultRow` — строка результата с персентилями (top-X%, gender percentile, category rank)
- **D2**: Таблица Y-o-Y прогресса (год, время, место, персентиль, динамика ▲/▼)

Данные уже приходят с бэкенда (`percentile`, `gender_percentile`, `category_rank`, `category_total`).

## Profile History (История изменений профиля)

**Приоритет:** Low
**Сложность:** Medium
**Добавлено:** 2026-01-24

### Проблема

Сейчас при пересчёте профиля старые значения **затираются** новыми. Нет возможности:
- Увидеть как менялся профиль со временем
- Понять почему вдруг изменились значения (например, walk_threshold прыгнул с 25% до 15%)
- Откатиться к предыдущей версии

### Решение

Добавить таблицу `profile_snapshots`:

```sql
CREATE TABLE profile_snapshots (
    id SERIAL PRIMARY KEY,
    user_id UUID REFERENCES users(id),
    profile_type VARCHAR(20),  -- 'hiking' | 'running'
    snapshot_data JSONB,       -- полный snapshot профиля
    activities_count INT,
    splits_count INT,
    calculated_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW()
);
```

### Что это даст

1. **Отладка** — понимать почему значения изменились
2. **Аналитика** — графики прогресса пользователя
3. **Откат** — возможность вернуться к предыдущей версии
4. **Прозрачность** — показывать пользователю "было → стало"

### Примечания

- Хранить последние N snapshots (например, 10) чтобы не раздувать базу
- Или хранить только при значительных изменениях (>10% разница в ключевых метриках)
