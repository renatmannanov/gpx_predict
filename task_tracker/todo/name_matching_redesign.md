## Что у нас сломано прямо сейчас

### Симптом

На локалке `Yuliya Kim` склеена в одного runner-а из **минимум пяти разных женщин**:
1985 (Alay.pro), 1991 (SkyRunGroup), 1982 (I LOVERUNNING Almaty), 1980, 1984 — плюс
4 фрагмента из AM-2025 без birth_year. Все 18 results привязаны к одному
`runner_id=8395`. На странице профиля смешаны результаты пяти разных бегуний.

### Корень

Парсер делает exact match только по `name_normalized`. Когда приходит вторая
`Yuliya Kim` (с другим birth_year), парсер **не различает** её от первой —
просто привязывает к существующему runner.

### Почему сейчас не "взрывается" на проде

На текущих CLAX-данных проблема меньше:
- 99.3% runners на проде имеют `birth_year`
- Имена в трейле редкие, тёзки попадаются нечасто
- Парсер всё равно создаёт ложные мёрджи, но в малых количествах
  (пример: `Pavlov Alexandr 2015` и `Aleksandr Pavlov 1984` лежат рядом
  и пока **повезло** что у них чуть разные `name_normalized`)

### Почему взрывается с AM

В AM 80% runners без `birth_year`. Имена кириллические, частые
(Сергей Иванов, Юлия Ким встречаются десятками). Без перепроектирования
матчинга нельзя залить ни одну AM-гонку — каждый Сергей Иванов будет
склеен в одного, страница профиля покажет смесь чужих гонок.

## Архитектурное решение: упрощение

### 1. Матчинг = пара `(name_normalized, birth_year)`

Парсер при импорте каждого result:
- Если **оба** известны (имя и birth_year), и в БД есть runner с такой парой → привязать
- Если **birth_year=NULL** или у существующих runner с таким именем все NULL →
  **создать нового runner** (фрагмент). Не угадывать.
- Никакой fuzzy логики при импорте — она только добавляет ложные мёрджи

### 2. UNIQUE constraint меняется

Было: `UNIQUE(name_normalized)` → не пускает второго Pavlov Alexandr
Будет: `UNIQUE(name_normalized, birth_year)` для известного birth_year +
       без unique для NULL (NULL≠NULL в Postgres, фрагменты могут дублироваться)

### 3. Тёзки = разные runners — это правильное состояние

Отец и сын Pavlov Alexandr 1984/2015 — два runners. Поиск показывает обоих
с пометкой года. На проф странице год не показывается (избыточно).

### 4. Юзер сам собирает свои фрагменты — UI ручного мёрджа

Если у `Sergey Ivanov` есть 3 фрагмента в БД (CLAX 1985, AM без birth, AM
другой год без birth) — юзер на странице "это тоже я?" видит кандидатов
по similarity и сам выбирает свои.

## Что делаем с фонетикой

Оставляем урезанный набор. Это компромисс между "все Сергеи под одним именем"
и "не склеить случайных тёзок 1985 г.р.".

### Оставляем (правила из текущего кода, проверенные)

| Правило | Покрывает |
|---------|-----------|
| `j$ → y` | Sergej → Sergey |
| `(гласная)j(?=гласная) → \1y` | Ajgul → Aygul |
| `(гласная)ev$ → \1yev` | Baev → Bayev |
| `(гласная)eva$ → \1yeva` | Baeva → Bayeva |
| `ii$ → iy` | Dmitrii → Dmitriy |
| `ei$ → ey` | Andrei → Andrey |
| `x → ks` | Alexandr → Aleksandr |
| `sch → shch` | Schavinskaya → Shchavinskaya |
| `ss → s` | Hassanov → Hasanov |

`ss → s` особенно важно — это **массовый артефакт КЗ-сервисов**: AM/Athletex
часто транслитерируют с двойным s там где паспортная норма с одинарным.
Хасанов, Юсупов, Иссанова, Расулов — десятки таких. Оставить обязательно.

### Убираем (риск ложных склеек)

| Правило | Почему убираем |
|---------|----------------|
| `h → kh` (standalone) | Ломает арабские имена: Hassan → Khassan, Hadji → Khadji |
| `ye → e` (start of word) | Ломает казахские: Yedil → Edil может быть разный человек чем Edil; Yelena/Elena — частая дилемма |

То что попадёт в "не нашлось точного совпадения" из-за выкинутых правил —
ловится UI ручного мёрджа. **Не критично**.

### Кириллица → латиница: оставляем

`Сергей Иванов` → `Sergey Ivanov`. Это нужно чтобы AM-данные могли
автоматически смёрджиться с CLAX, **если** birth_year известен в обеих
системах (что редко в AM, но бывает).

## Что мы НЕ делаем автоматически

### AM-кириллица ↔ CLAX-латиница без birth_year

В CLAX `Sergey Ivanov, 1985`. В AM `Сергей Иванов`, без birth_year.
Транслитерация даст `sergey ivanov` для AM. Но **birth_year отличается**
(NULL vs 1985), составной ключ не совпадёт. Это **остаётся фрагментом**.

Цена честности. Юзер сам сольёт через UI ручного мёрджа.

### Авто-разрешение коллизий разных людей

Если в БД лежит два `Pavlov Alexandr` (1984 и 2015), и приходит третий
без birth_year — **не привязываем ни к кому**, создаём нового фрагмента.

## Стратегия пересборки данных

Ничего user-related не трогаем. Race-данные пересобираем с нуля.

### Что пересобираем

- `runners` (TRUNCATE)
- `clubs` (TRUNCATE)
- `race_results` (TRUNCATE)
- `race_distances` (TRUNCATE)
- `race_editions` (TRUNCATE)
- `races` (TRUNCATE)
- `runner_name_aliases` (TRUNCATE)
- `club_name_aliases` (TRUNCATE)

### Что НЕ трогаем

- `users`, `strava_tokens`, `strava_activities`, `strava_activity_splits`
- `user_performance_profiles`, `user_run_profiles`, `profile_snapshots`
- `gpx_files`, `notifications`, `strava_sync_status`
- Alembic versions, индексы, constraints

### Workflow

1. Полностью пересобираем локально с нуля (новый код, парсинг всех гонок
   из catalog.yaml).
2. Делаем pg_dump прод-БД (полный) как backup.
3. На проде TRUNCATE только race-таблицы (см. список выше).
4. COPY race-таблицы локалка → прод.
5. Smoke test: поиск, профиль, гонка, percentile.

При плохом результате — откат через pg_dump.

## UI ручного мёрджа — отдельная задача

Не блокирует пересборку, но без неё юзеры не смогут собрать AM-фрагменты
своих результатов в один профиль. Состоит из:

### Backend API (нужно добавить)

- `GET /runners/{id}/merge-candidates` — кандидаты с similarity, метаинфой,
  результатами для disambiguation
- `POST /runners/{id}/merge` — выполнить мёрдж (использует существующую
  функцию `merge_runner_pair`)
- `GET /results/search?bib=X&race=Y&distance=Z` — поиск result по
  стартовому номеру
- `POST /runners/{id}/attach-result` — добавить найденный result в профиль

### Frontend (нужно добавить)

- На странице runner profile — секция "Это тоже я?" с кандидатами
- Поиск result по bib + кнопка "добавить в мой профиль"
- Опционально: Strava auto-suggest — если есть активность в день гонки
  на нужное расстояние, предложить мёрдж

### Защита от ложного мёрджа

Не добавлять никаких преград "просто чтобы было". Реалистично:
- **Bib + race + distance** — основной идентификатор
- **Strava activity** — bonus tier для тех у кого подключена. Можно автоматически
  предлагать соответствие, юзер подтверждает
- **Не делать** проверку "финишное время" (избыточно, bib достаточно)
- **Не делать** периодические рассылки

Кейс с тёзками в одной гонке (два `Sergey Ivanov 1985` с одинаковым bib —
в природе невозможно, bib уникален в (race, distance)).

| Что | Почему |
|-----|--------|
| `_fuzzy_match_runner` из импорта | Тёзки теперь идут в отдельные runners — fuzzy не нужен при импорте |
| `compute_new_sim.py` | Аудит-скрипт отыграл своё |
| `fix_runner_clubs.py` | One-time фикс отыграл своё |
| `update_phonetic_norm.py --detect/--merge/--update` фазы | Раз UNIQUE по паре `(name, birth)`, коллизий не будет, нужен только пересчёт |
| 2 фонетических правила (`h→kh`, `ye→e` start) | Слишком рискованны для KZ-кейсов |
