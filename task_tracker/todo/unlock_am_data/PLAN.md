# Разлочить данные Алматы Марафон + почистить дубли Athletex

> Статус: pending
> Дата: 2026-06-08
> Тип: фича + фикс данных

## Цель

Залить данные almaty-marathon.kz на портал ayda.run, попутно вылечив существующие
дубли athletex (склейки разных людей под одним runner). Ключевое решение —
**изоляция источников через поле `source`**: AM-бегуны и athletex-бегуны никогда
не сопоставляются между собой. Матчинг внутри источника — строго по паре
`(name_normalized, birth_year)`, без fuzzy-угадывания.

## Решения пользователя (зафиксированы)

1. **Изоляция через `source`** у runner (`am` / `athletex`). Матчинг только внутри своего источника.
2. **Без межисточниковой транслитерации** — кириллица AM не сопоставляется с латиницей athletex.
   Транслитерация остаётся только для поиска по-русски внутри AM.
3. **Веб без авторизации** — UI ручного мёрджа и личный кабинет откладываются.
4. **Фрагменты без birth_year остаются разрозненными** — это осознанный компромисс честности.

## Что НЕ входит (отложено)

- UI ручного мёрджа («это тоже я») + веб-авторизация
- Авто-загрузка гонок по расписанию
- Перенос PredictPage (прогноз по GPX) в веб
- Урезание фонетики `h→kh`/`ye→e` — НЕ трогаем (при изоляции source риск ложных
  склеек между источниками устранён архитектурно, внутри источника эти правила
  скорее помогают)

## Контекст: почему так

- `KIM YULIYA` (runner 8395) уже склеена из 5 разных женщин (1980/1982/1984/1985/1991),
  `Pavlov Alexandr` — из 3 разных людей. Это **athletex-данные, ещё до всякого AM**.
  Лечится пунктом «матчинг по (имя, год)» при пересборке.
- AM: 80% без birth_year, кириллица, частые тёзки → межисточниковое сопоставление
  ломает всё → изолируем по source.
- `source` уже передаётся в `save_to_db()` (значения `clax` / `almaty-marathon`),
  но **не сохраняется на уровне runner** — колонки нет. Это добавляем.
