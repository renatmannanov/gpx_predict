# R&D Фаза R3: Buddy Matching — найди напарника для маршрута

## Цель

Матчинг бегунов/хайкеров по совместимости на основе реальных GPS-данных (gradient-pace profile), а не self-reported анкет. Начать внутри клубов (группировка на лонгах), расширить до поиска напарника вне клубов.

---

## Контекст и мотивация

### Проблема

1. **Внутри клубов:** На клубных лонгах группа растягивается по маршруту. Чем больше людей, тем сильнее разброс. Люди теряют друг друга, новички отстают, быстрые убегают. Тренер не может разбить на группы без объективных данных.

2. **За пределами клубов:** Люди ищут напарников для гор. В одиночку — опасно (нет связи, травмы, погода). Существующие решения не работают (см. ресерч рынка ниже).

3. **Readiness flow:** Цепочка "Маршрут → Грейд → Готов ли ты?" сейчас обрывается на тупике. "Найди напарника" превращает тупик в следующее действие:
   - **Готов** → "Время: 5:30. Скачай GPX, удачи!"
   - **С оговорками** → "Рекомендуем не в одиночку. Найди напарника →"
   - **Не готов** → "Попробуй маршрут полегче. Найди напарника для него →"

### Ресерч рынка: "найди напарника для бега"

**Ни один существующий продукт не решает эту задачу хорошо:**

| Продукт | Модель | Проблема |
|---------|--------|----------|
| **JoggingBuddy.com** (с 2009) | Доска объявлений: анкета + локация | Self-reported данные. "Бегу 6:00/км" — а на самом деле 7:30. Нет матчинга по данным, нет Strava. Масштаб крошечный |
| **Running Mate** (getrunningmate.com) | "Uber для бегунов" — safety для женщин | Платные верифицированные напарники. Не матчинг по уровню, а безопасность. Только США |
| **Jaha** (~2018) | "Tinder для бегунов" | Мёртвый проект. Сайт не найден |
| **DePassport** | Gym buddy | Не специфично для бега |
| **Strava** | Social features (follow, clubs) | Сознательно НЕ делает matching. "Strava has GPS data to suggest nearby friends by location and ability, but they don't" (LetsRun). Privacy concerns |

**Общая проблема всех:** Self-reported данные. Никто не использует реальные GPS/Strava данные для матчинга. Это наш gap.

### Наше преимущество

У нас уже есть:
- **Gradient-pace profile** (`gradient_pace_profile JSONB` в users) — как человек реально бежит на подъёмах, спусках, плоскости. Двое с похожим профилем = комфортно вместе
- **Route matching** (R1) — бегали ли они этот маршрут или похожие
- **Activity patterns** — когда обычно бегают (будни/выходные, утро/вечер), недельный объём
- **Community Application exception** (Strava) — легально показывать computed данные внутри клуба

Матч = близкий gradient-pace profile + пересечение маршрутов + совпадение расписания. Не "вы оба написали что бегаете средне", а объективные данные.

## Стратегия: от клубов к городу

### Почему НЕ intent board (доска "планирую пойти")

Intent board ("кнопка → список людей на дату") работает только при критической массе. На 42 юзерах кнопку нажмут 0-2 человека — пустой экран убивает доверие к фиче.

### Почему начинать с клубов

- Люди **уже бегают вместе** → ниже барьер доверия
- Тренер **видит проблему** ("растягиваются") → pain point подтверждён
- **Не нужна критическая масса** — в SRG ~20 человек, достаточно для группировки
- **Community App exception** → показывать данные участников друг другу легально
- **Не нужен opt-in** — участники клуба уже в системе

### Три ступени

**Ступень 1: Группировка внутри клуба (Фаза 6-7)**

Тренер создаёт лонг → система предлагает группы по темпу:

```
Лонг в субботу: Кок-Жайляу
Участники: 12 человек

Рекомендованные группы по темпу на подъёмах:
🔴 Быстрая (4:30-5:30/км): @user1, @user3, @user7
🟠 Средняя (5:30-7:00/км): @user2, @user5, @user8, @user11
🟢 Комфортная (7:00+): @user4, @user6, @user9, @user10, @user12
```

Это решает реальную проблему тренера, валидирует алгоритм на живых данных, не требует новой инфраструктуры.

**Ступень 2: Межклубный матчинг (после 50+ юзеров)**

Юзер из SRG видит совместимых бегунов из RUNFINITY или без клуба. Кросс-опыление: "свободные" бегуны знакомятся с участниками клубов, приходят в клубы.

Opt-in обязателен. Профиль показывает: имя, грейд, "бегал этот маршрут X раз", совместимость по темпу. НЕ показывает: конкретные активности, GPS-треки, даты.

**Ступень 3: Публичный поиск напарника (масштаб)**

Полноценная фича на портале. Привязана к маршрутам и readiness flow. Расширение за пределы клубов.

## Алгоритм матчинга

### Входные данные (уже есть в системе)

| Данные | Источник | Описание |
|--------|----------|----------|
| Gradient-pace profile | `users.gradient_pace_profile` | 11 бакетов от -30% до +30%, медианный темп в каждом |
| Маршруты | `activities.matched_route_id` (R1) | Какие маршруты бегал, сколько раз, лучшее время |
| Паттерны | `activities.date` | Дни недели, время суток |
| Объёмы | `activities.distance_m` | Недельный километраж, набор высоты |

### Расчёт совместимости

```
compatibility_score = w1 * pace_similarity + w2 * route_overlap + w3 * schedule_match + w4 * volume_ratio

pace_similarity:    cosine similarity между gradient-pace profiles
route_overlap:      Jaccard index маршрутов (пересечение / объединение)
schedule_match:     совпадение дней и времени (из паттернов активностей)
volume_ratio:       1 - |vol_A - vol_B| / max(vol_A, vol_B)
```

Для ступени 1 (клубы) достаточно только `pace_similarity` — кластеризация по темпу на подъёмах для конкретного маршрута.

### Кластеризация для клубных лонгов (ступень 1)

Упрощённый вариант — не полный matching score, а группировка по одному параметру:

1. Для маршрута лонга → определить ключевые градиенты (средний подъём, средний спуск)
2. Для каждого участника → достать его темп на этих градиентах из gradient-pace profile
3. K-means или простые пороги → 2-4 группы

## Privacy и Strava compliance

### Opt-in

- **Ступень 1 (клубы):** Opt-in не нужен — участники клуба уже согласились на Community Application
- **Ступень 2-3 (межклубный/публичный):** Opt-in по умолчанию **OFF**. Явное включение "Я ищу напарника" в настройках

### Что показываем другим юзерам

- Имя, фото (из Strava/Telegram)
- Грейд уровня (🟢🟡🟠🔴)
- "Этот маршрут бегал(а) N раз"
- Совместимость по темпу: "Ваши темпы отличаются на ~30 сек/км"
- Preferred days (из паттернов)

### Что НЕ показываем

- Конкретные активности, GPS-треки, пульс, даты пробежек
- Strava профиль, телефон, email — до взаимного согласия

### Strava API compliance

Мы не "показываем данные юзера А юзеру Б". Мы показываем **computed совместимость**. Конкретные Strava Data не отображаются. Community Application exception покрывает клубный сценарий.

Для Garmin-данных (`source: 'garmin'`) — ограничений на агрегацию нет.

## Safety-угол (горы Алматы)

В контексте гор это не просто "мотивация бегать вместе". Это безопасность:

- Люди гибнут на маршрутах в одиночку
- Новички идут неподготовленными
- Нет мобильной связи на многих тропах
- Группа = безопаснее

"Не ходи один на Фурманова. Вот 3 человека твоего уровня которые тоже планируют в эти выходные" — это safety feature, не dating app.

## Кросс-опыление клубов (побочный эффект)

Юзер из SRG видит совместимого напарника из RUNFINITY или вообще без клуба. Это:
- Расширяет социальный граф за пределы одного клуба
- "Свободные" бегуны приходят в клубы через знакомство с участниками
- Укрепляет позицию ayda как платформы, а не инструмента одного клуба
