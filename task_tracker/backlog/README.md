## A. Точность предсказаний (ядро продукта)

Связанные задачи, общая цель — снизить ошибку предсказаний.

### A1. Калибровка профилей + Strava Streams/FIT
> [calibration_backlog.md](calibration_backlog.md) + [backlog_strava_streams.md](backlog_strava_streams.md)

Две стороны одной проблемы:
- **Калибровка:** персонализированный метод даёт +35.8% ошибку. Причина — один градиентный бакет смешивает асфальт и скрамблинг (12x разброс пейса). Решение: фильтрация пеших сегментов, медиана, тонкие категории
- **Streams/FIT:** текущие 1km-сплиты смешивают рельеф → неточная классификация. Streams API даёт посекундные данные. FIT-файлы с часов обходят лимиты Strava. Ожидание: MAPE 14.5% → 10-12%
- **Проблема Strava:** API не различает Run vs Trail Run → нельзя автоматически разделить профили по типу поверхности

Можно делать **параллельно или последовательно** — не блокируют друг друга.

### A2. Hiking-профили — IQR + effort levels + рефакторинг
> [backlog_hiking_profile.md](backlog_hiking_profile.md) + [backlog_hiking_service_refactor.md](backlog_hiking_service_refactor.md) + [backlog_hiking_profile_extraction.md](backlog_hiking_profile_extraction.md)

Три связанные задачи:
1. **IQR + effort levels** — применить к хайкингу то, что работает для trail running. Текущая ошибка: -49.6%
2. **HikingService рефакторинг** — монолитный `PredictionService` → полноценный `HikingService` (как `TrailRunService`)
3. **Извлечение UserHikingProfile** — техдолг, вынести в отдельный файл

**НЕ блокируется калибровкой (A1).** Можно сделать на текущих данных, а потом сравнить с улучшенными профилями из A1. Сохраняем текущие результаты для сравнения.

### A3. Разделение Road / Trail профилей
> [calibration_backlog.md](calibration_backlog.md) — раздел "Рекомендация #6"

Единый run-профиль смешивает асфальт и горы. Нужны раздельные профили: Road Run (парк, асфальт) и Trail Run (горы, техничный). Блокер: Strava API не различает тип — нужна либо ручная разметка, либо Garmin/FIT-интеграция.

**Зависит от:** A1 (Streams/FIT дадут данные для разделения)

## B. Портал ayda.run — магнит для пользователей

Портал — основной канал привлечения. Эти задачи делают его полезнее и заметнее.

### B1. SEO — мета-теги для SPA
> [backlog_seo_meta.md](backlog_seo_meta.md)

React SPA не генерирует мета-теги → поисковики и мессенджеры не видят контент. Варианты: react-helmet (простой, бесполезен для ботов), prerender.io (быстрый, платный), SSR-миграция (тяжёлый).

### B2. Races MVP — доделки портала
> [races_mvp.md](races_mvp.md)

- Disambiguation для одинаковых имён (нужно при больших гонках)
- Алиасы дистанций из YAML вместо хардкода
- Персонализированный hiking-прогноз для гонок

### B3. Качество данных — клубы и профили
> [BACKLOG_2.md](BACKLOG_2.md)

- 6 неоднозначных пар клубов (Railways Club vs Railways Team и т.д.) — ручная проверка
- Profile history (snapshots) — JSONB-снапшоты профилей для отладки и "было → стало"

### B4. YAML + GPX хранение — масштабирование каталога
> [races_yaml_and_gpx_storage.md](races_yaml_and_gpx_storage.md)

Один `races.yaml` не масштабируется. Решение: `catalog/*.yaml` (один файл на гонку). GPX: `gpx/{race_id}/{distance}.gpx`. Триггер: >10 гонок.

### B5. Мелкие UX/tech debt
> [backlog_1.md](backlog_1.md)

- Показывать статус Strava на странице хайкинга
- Strava sync: начинать с новейших активностей
- Убрать debug-логирование

## C. Cross-service (ayda_run <> gpx-predict)

### C1. Предикт маршрута лонга из ayda_run
> [races_mvp.md](races_mvp.md) — пункт #5 (internal predict API)

В ayda_run у каждой гонки вложен GPX-файл. Можно дать пользователям предсказание времени прямо из бота — "Хочу пробежать Tengri Ultra VK1000, сколько займёт?". API: `POST /api/v1/internal/predict`. Cross-service интеграция уже есть.

### C2. User matching — подбор партнёров
> [user_matching.md](user_matching.md)

Поиск совместимых тренировочных партнёров по профилю (flat_pace, градиентный профиль, объём). Read-only сервис. Требует opt-in. Cross-service интеграция уже есть, блокер — достаточное количество пользователей с профилями.

## D. Долгосрочные / исследовательские

### D1. Лавинный анализ маршрутов
> [avalanche_risk_analysis.md](avalanche_risk_analysis.md)

Оценка лавинной опасности по GPX-маршруту (Иле-Алатау). Анализ углов склонов, ATES-классификация, интеграция с ingeo.kz. Уникально для Казахстана. Планируется: зима 2026-2027.

### D2. Погода и тип поверхности

Нет отдельного файла — записать при планировании. Включает:
- **Погода:** температура, ветер, осадки влияют на время. Интеграция с метео-API
- **Тип поверхности:** асфальт, грунт, снег, лёд — разный коэффициент скорости. Требует разметки маршрутов или автоматического определения (OSM?)
- **Связь с A3:** разделение Road/Trail профилей — первый шаг к учёту поверхности
