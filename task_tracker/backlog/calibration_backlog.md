## Проблема персонализированного метода

### Причина ошибки

Персонализированный метод использует средний pace по категориям градиента из профиля пользователя. Проблема в том, что **в одной категории смешиваются разные типы активностей:**

- Асфальтовые спуски (быстрый бег)
- Технические горные спуски (медленная ходьба/скремблинг)

### Пример: категория Steep Down (gradient < -10%)

**Данные из профиля:** 9.80 min/km (среднее по 19 samples)

**Реальный разброс в этой категории:**

| Самый быстрый | Самый медленный |
|---------------|-----------------|
| 4.47 min/km | 55.70 min/km |
| Povlen Trail 2025 (гонка, km 24) | Lunch Trail Run (технический участок) |

**Разница в 12 раз при одинаковом градиенте!**

### Детальный анализ категории -10% to -12%

| Splits | Avg Pace | Тип активностей |
|--------|----------|-----------------|
| Все 18 | 11.12 min/km | Смесь всего |
| Без ходьбы (pace ≤ 12) | **7.77 min/km** | Только бег |
| Только быстрые (pace ≤ 8) | **5.99 min/km** | Асфальт/гонки |

**Вывод:** Исключение всего 3 сплитов из 18 (ходьба) меняет средний pace с 11.12 до 7.77 min/km — разница 3.35 min/km!

### Конкретные примеры (grade -10%)

**Быстрые спуски (бег):**
| Pace | Активность | Km | Почему быстро |
|------|-----------|-----|---------------|

| 5.71 | Morning Asphalt Sky Run | 14 | Асфальт |
| 5.88 | Morning Asphalt Sky Run | 16 | Асфальт |

**Медленные спуски (ходьба):**
| Pace | Активность | Km | Почему медленно |
|------|-----------|-----|-----------------|
| 36.86 | Bayankol #1 Trail Run | 6 | Технический горный спуск |

## Рекомендации по улучшению

### 1. Фильтрация ходьбы при расчёте профиля

**Проблема:** Сегменты с pace > 12-15 min/km — это не бег, а ходьба/скремблинг.

**Решение:** При расчёте профиля исключать сегменты с pace выше порога.

```python
# Примерная логика
WALKING_THRESHOLD_MIN_KM = 12.0  # или динамически от flat_pace * 2

def should_include_segment(pace, gradient):
    if pace > WALKING_THRESHOLD_MIN_KM:
        return False  # Это ходьба, не бег
    return True
```

**Ожидаемый эффект:** Для steep down pace изменится с 9.80 → ~7.5-8.0 min/km

### 2. Разделение профилей по типу поверхности

**Проблема:** Асфальт и технический trail — разные виды бега.

**Варианты решения:**

a) **Два профиля:** road/easy trail vs technical trail
   - Требует классификации активностей (сложно автоматизировать)

b) **Использовать медиану вместо среднего**
   - Менее чувствительна к выбросам
   - Простая реализация

c) **Percentile-based approach**
   - Использовать 25-й или 75-й перцентиль в зависимости от типа прогнозируемого маршрута

### 3. Более детальная градация градиентов

**Текущие категории:**
- Steep down: < -10%
- Moderate down: -5% to -10%
- и т.д.

**Проблема:** Категория "steep down" слишком широкая (-10% и -40% — разные вещи).

**Решение:** Добавить категории:
- Very steep down: < -20% (часто ходьба)
- Steep down: -10% to -20% (бег возможен)

### 4. Адаптивный walk threshold

**Текущий подход:** Фиксированный порог (сейчас не используется).

**Улучшение:** Порог = flat_pace * коэффициент

```python
walk_threshold = flat_pace * 1.8  # Если flat_pace = 6.85, то порог = 12.3 min/km
```

## Backlog: Новые идеи (2026-02-13)

### 5. PACE_MAX_THRESHOLD_RUN — возможно убрать полностью

**Контекст:** `PACE_MAX_THRESHOLD_RUN` был тестовым значением (15.0 min/km), отрезал ~90% steep uphill данных. В Phase 1 увеличен до 45.0 min/km как санитарный порог.

**Доказательство:** В baseline профиле steep_up = 14.56 min/km (7 samples), а после пересчёта на всех данных raw mean = 21.87 (66 samples). Порог 15.0 отрезал большинство горных данных.

**Текущее решение (Phase 1):** Увеличен до 45.0 min/km — пропускает все нормальные данные, фильтрует только явные остановки/ошибки GPS.

**Будущее:** После обкатки IQR, возможно убрать верхний порог полностью (IQR адаптивно фильтрует в каждой категории). PACE_MIN_THRESHOLD_RUN = 2.5 min/km оставить.

**Приоритет:** Низкий. Текущее значение 45.0 достаточно.

### 6. Разделение профилей Road Run / Trail Run

**Проблема:** Единый беговой профиль смешивает road runs и trail runs. Flat pace на асфальте и грунте отличается, на подъёмах тоже разная техника. Effort levels частично компенсируют, но не устраняют корневую причину.

**Идея:** Три отдельных профиля пользователя:
- **Hiking profile** — как сейчас (Walk + Hike активности)
- **Road Run profile** — только road/city runs (асфальт, парк)
- **Trail Run profile** — только trail runs (горы, грунт, техничный рельеф)

Каждый со своими gradient_paces, gradient_percentiles, effort levels.
При предсказании пользователь выбирает тип маршрута → используется соответствующий профиль.

**Сложность:** Высокая. Нужно:
1. Надёжно классифицировать активности на road/trail
2. Отдельная модель или подтип UserRunProfile для каждого
3. Два набора gradient_paces/gradient_percentiles
4. UI/UX для выбора типа маршрута при предсказании

**Текущее ограничение (2026-02-14):** Strava не разделяет Run/TrailRun автоматически — у тестового пользователя все 107 беговых = `Run`, ноль `TrailRun`. Классификация по elevation (эвристика) возможна, но ненадёжна. Часы Garmin/Coros/Suunto содержат surface type в метаданных — с ними разделение будет точным.

**Решение:** Отложить до интеграции Garmin/других часов.

**Приоритет:** Средний. Реализовать после Garmin integration.
