# Матчинг пользователей по профилю

**Статус:** бэклог
**Зависимости:** cross_service_integration (шаги 1-2.5), стабильная предикт-модель
**Контекст обсуждения:** 2026-02-26

---

## Цель

Подбор пар/групп пользователей по уровню подготовки для совместных тренировок и забегов.
Матчинг делает **gpx_predictor** (не ayda_run) — он владеет профилями.

---

## Три направления матчинга

| Направление | Профиль в системе | Готовность данных |
|---|---|---|
| Trail running | UserRunProfile (Run + TrailRun + VirtualRun) | есть |
| Hiking | UserHikingProfile (Hike + Walk) | есть |
| Road running | нет отдельного профиля — данные смешаны в UserRunProfile | частично |

**Road running:** отдельного шоссейного профиля нет. Run profile включает trail + road.
Варианты: (a) использовать flat_pace из run profile как есть, (b) разделять trail/road при sync — усложнение, не для v1.

---

## Критерии матчинга

### Используем в v1 (данные уже есть в профилях)

| Критерий | Что есть | Роль в матчинге |
|---|---|---|
| **Flat pace p50** | `gradient_percentiles["flat_3_3"]["p50"]` | Основная ось — "базовая скорость" |
| **Pace по грейдам** | `gradient_percentiles` (11 категорий) | Cosine similarity — два бегуна с одним flat pace могут кардинально различаться на горках |
| **Стабильность (p25-p75)** | `gradient_percentiles[cat]["p25"/"p75"]` | Узкий разброс = стабильный бегун; стабильный + нестабильный = плохая пара |
| **Vertical ability** | `vertical_ability` (hiking only) | Способность на подъёмах: <1.0 = сильнее стандарта |
| **Walk threshold** | `walk_threshold_percent` (run only) | Градиент перехода в ходьбу |
| **Объёмы** | `total_distance_km`, `total_elevation_m` | Фильтр "серьёзности": 50 км/мес vs 300 км/мес — разные цели |
| **Кол-во активностей** | `total_activities` | Фильтр надёжности профиля |

### Отложено (нет данных или сложно)

| Критерий | Проблема | Решение |
|---|---|---|
| **Пульсовые зоны** | avg_heartrate есть в splits, но не агрегируется; зоны индивидуальны (нужен max HR / LTHR) | Можно добавить avg_heartrate в профиль, но зоны — отдельная задача |
| **Локация** | `start_latlng` в Strava activity, но мы не сохраняем (PII) | Задача ayda_run (чаты по городам) |
| **Время суток** | `start_date` есть в activities, можно извлечь паттерн | Скорее задача ayda_run (организация тренировок) |
| **Тренд формы** | Сравнение pace за последние 3 мес vs предыдущие | Интересно, но сложно и не для v1 |
| **Дистанция забегов** | Спринтер (5-10км) vs ультра (50км+) — из avg/max distance | Можно, но не приоритет |
| **Тип активностей (доля)** | 80% TrailRun = трейлраннер, 50/50 = мультиспортсмен | Можно считать из activity_type |

## Архитектурный подход

### Ключевое решение: матчинг НЕ усложняет предикт-модель

```
Strava → Sync → Profile (уже работает, НЕ трогаем)
                    ↓
              MatchingService (новый read-only слой)
```

**Почему так:**
- Предикт-модель ещё не протестирована на пользователях (только на одном)
- Нельзя усложнять то, в чём не уверены
- Матчинг только ЧИТАЕТ профили, не влияет на их расчёт
- Если профиль окажется неточным → фиксим профиль, матчинг автоматически улучшится

## Приватность

**Требование:** пользователь должен дать явное согласие на участие в матчинге.

Нужно добавить:
- Поле `allow_matching: bool = False` в User модель
- Команда/кнопка в боте для включения/выключения
- Batch endpoint отдаёт ТОЛЬКО пользователей с `allow_matching=True`

## Предварительный алгоритм (v1)

```
MatchingService (read-only):

  Input:  telegram_id + type (trail_run | hiking)
  Output: список совместимых пользователей с score

  Алгоритм:
  1. Загрузить профиль целевого юзера
  2. Загрузить профили всех юзеров с allow_matching=True
  3. Для каждой пары считать similarity:
     - 60% — flat_pace_p50 distance (основной фактор)
     - 25% — gradient profile similarity (cosine по 11 категориям)
     - 15% — volume similarity (total_km, нормализованный)
  4. Фильтры:
     - min_activities > 5 (профиль ненадёжный)
     - min_sample_count > 10 для flat категории
  5. Вернуть top-N с scores + confidence
```

## Сложности и риски

### Определение "level"
- В плане cross_service_integration ответ содержит `"level": "intermediate"`, но в коде нет системы классификации
- Пороги субъективны: flat pace 6:00/km = "intermediate" для hiking, но "beginner" для бега
- Пользователи могут обижаться на ярлыки
- **Решение:** Либо не включать level, либо использовать нейтральные названия

### Нет flat данных
- Горный бегун может не иметь flat сплитов → `flat_3_3` пустой
- Нужен fallback на `avg_flat_pace_min_km` (legacy) или ближайшую категорию

### gradient_coverage
- В примере response `"gradient_coverage": 11` — такого поля нет, нужно вычислять
- Вопрос порога: категория "покрыта" при sample_count > 0? > 5? > 20?

### Batch производительность
- При текущем масштабе (десятки юзеров) — не проблема
- При сотнях — нужна пагинация и/или кэширование
- Профили меняются редко (после sync) — можно кэшировать на часы

### Road running
- Нет отдельного профиля — trail и road смешаны в UserRunProfile
- Разделение потребует изменений в sync (усложнение модели) — не для v1

## API (черновик)

```
# Один профиль
GET /api/v1/internal/profile?telegram_id={id}&type=trail_run
→ {telegram_id, type, has_profile, flat_pace_p50, total_activities, ...}

# Batch профилей
POST /api/v1/internal/profiles/batch
Body: {telegram_ids: [...], type: "trail_run"}
→ [{telegram_id, ...}, ...] (только allow_matching=True)

# Матчинг (gpx_predictor считает similarity)
POST /api/v1/internal/match
Body: {telegram_id: "123", type: "trail_run", top_n: 10}
→ [{telegram_id, similarity_score, confidence, flat_pace_p50, ...}, ...]
```
