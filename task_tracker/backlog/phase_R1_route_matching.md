# R&D Фаза R1: Route Matching Engine

## Цель

Автоматически определять, какой маршрут из каталога (фаза 8) пробежал юзер, по GPS-треку его Strava/Garmin активности.

## Контекст

Из agent_brief.md (секции 2, 5):

Route matching — ключевая фича для персонального слоя:
- "Ты бегал Кок-Жайляу 5 раз, лучшее время 2:38"
- Автоматическое наполнение "Твои маршруты"
- Прогресс по маршрутам

**Отложен из V1** — для MVP хватит ручной привязки. R&D задача.

## Проблемы (из agent_brief.md секция 0)

- **GPS drift на трейлах:** 10-30м отклонение — обычное дело
- **Out-and-back:** Один и тот же путь туда и обратно
- **Разные точки старта:** Люди начинают с разных мест
- **Частичные прохождения:** Юзер пробежал только часть маршрута
- **Разные варианты маршрута:** Обход через другую тропу

## Алгоритм (из agent_brief.md)

1. Декодирование polyline из Strava activity
2. Сравнение GPS-трека с эталонными маршрутами из каталога
3. Fuzzy matching: порог ~80% точек трека в пределах 50м от эталона
4. Результат: `matched_route_id` или NULL

## Данные

**Вход:** `polyline` из таблицы activities (Strava backfill)
**Эталон:** GPX файлы из `content/routes/gpx/`
**Выход:** `matched_route_id` в таблице activities

## Зависимости

- Фаза 8 (каталог маршрутов с GPX)
- Strava backfill через ayda_run (polyline в activities)

## Когда делать

После каталога маршрутов (фаза 8) и когда будет достаточно юзеров с backfill для тестирования. Можно начать с Рената — у него есть история горных активностей.
