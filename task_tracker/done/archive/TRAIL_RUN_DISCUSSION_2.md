## Найденные противоречия (Claude)

После изучения TRAIL_RUN_DISCUSSION.md и кода предикторов, выявлено **12 ключевых противоречий**:

### 1. Tobler показывается как отдельное время, но считается только для HIKE сегментов

**Проблема:** В списке времён показывается `tobler: 18мин`, но это время **только для 0.4 км ходьбы**, а не для всего маршрута 20 км.

**Почему это плохо:**
- Пользователь видит "tobler: 18мин" рядом с "Strava GAP: 3ч 50мин" и не понимает, что это за число
- Создаёт ложное впечатление, что tobler — это альтернативный расчёт для всего маршрута
- В hiking tobler показывает время для ВСЕГО маршрута, а здесь — только для части

**Что должно быть:** Либо убрать tobler из списка времён, либо показывать с пояснением "(только ходьба: 0.4 км)"

### 2. Несогласованность "primary_time" — неясно какой метод используется

**Проблема:** `primary_time` выбирается по сложной логике:
```
RUN: run_personalized → strava_gap → minetti_gap
HIKE: hike_personalized → tobler
```

Но пользователю не показывается, какой именно метод был выбран для каждого сегмента.

**Почему это плохо:**
- Пользователь видит "Комбинированный: 4ч 8мин", но не знает из чего это складывается
- Если персонализация есть только для части категорий градиента — будет mix методов
- В hiking нет такого понятия — там всегда показывается конкретный метод

### 3. Fatigue применяется ко ВСЕМ методам в totals, включая tobler

**Проблема в коде** (service.py:339-350):
```python
for method, time in times.items():
    fatigue_adj, _ = self._fatigue_service.apply_to_segment(...)
    totals[method] += fatigue_adj
```

**Почему это плохо:**
- Strava GAP и Minetti GAP уже включают fatigue в свои totals
- Tobler (научная формула ходьбы) тоже получает fatigue, хотя это не имеет смысла для ходьбы
- Пользователь видит "Strava GAP: 3ч 50мин", но это уже С fatigue — неочевидно

### 4. Порог ходьбы 25% слишком высокий — почти всё считается бегом

**Проблема:** По умолчанию `walk_threshold = 25%`. На маршруте с 2322м набора только 2% (0.4 км) — ходьба.

**Почему это плохо:**
- На градиенте 20% реально бежать невозможно, но система считает это бегом
- Hiking предиктор не имеет такого порога — весь маршрут считается ходьбой
- Реальные бегуны переходят на ходьбу при 15-18% подъёма

**Что должно быть:** Порог по умолчанию 15-18%, или динамический на основе данных пользователя

### 5. "Персональный (бег)" vs "run_personalized" — разные вещи показываются

**Проблема:**
- В сообщении показывается "Персональный (бег): 3ч 50мин"
- Но в коде есть **два** персонализированных метода: `run_personalized` и `hike_personalized`
- "Персональный (бег)" — это только RUN сегменты, а HIKE сегменты считаются по другому

**Почему это плохо:**
- Пользователь думает что "Персональный (бег)" — это весь маршрут
- На самом деле 0.4 км (ходьба) посчитаны по другому методу
- В hiking нет такого разделения

### 6. Нет rest time и lunch break для trail running

**Проблема:** Hiking добавляет:
- Rest time: 10-15 мин на час в зависимости от опыта
- Lunch break: 30 мин если >4 часов
- Safe time: +20%

Trail running не добавляет ничего из этого.

**Почему это плохо:**
- На 4-часовом трейле реально нужны питпоинты и отдых
- Особенно на ultra (50km+) — без отдыха время нереалистичное
- Прогноз выглядит оптимистичнее hiking, хотя trail run физически тяжелее

### 7. Множители (experience, backpack, group) не применяются к trail running

**Проблема:** Hiking использует multipliers:
- Experience: 0.85 - 1.5x
- Backpack: 1.0 - 1.25x
- Group size: 1.0 - 1.3x
- Altitude: 1.0 - 1.35x

Trail running не использует ни один из этих множителей.

### 8. "Комбинированный" включает fatigue, но другие методы — тоже

**Проблема:** В описании написано:
> Комбинированный = primary_time + fatigue

Но в коде ВСЕ методы в totals включают fatigue. Разница между "strava_gap" и "combined" только в выборе primary_time для некоторых сегментов.

### 9. Нет round_trip логики для trail running

**Проблема:** Hiking умеет считать туда-обратно:
- Меняет местами подъём и спуск
- Даёт 10% скидку на знакомый маршрут

Trail running не имеет такой опции.

### 10. Персонализация hiking vs trail run — разные источники данных

**Проблема:**
- **Hiking**: использует `UserHikingProfile` из Strava Hike активностей
- **Trail Run**: использует `UserRunProfile` из Strava Run/TrailRun активностей

Но на HIKE сегментах trail run использует `hike_personalized` из `UserHikingProfile`.

**Почему это смущает:**
- Пользователь может не иметь hiking активностей, но иметь run
- Тогда RUN сегменты персонализированы, а HIKE — нет (fallback на tobler)
- Результат — mix персонализации и научных формул

### 11. GAP режимы (Strava vs Minetti) — зачем два?

**Проблема:** Пользователь выбирает режим GAP, но:
- Оба показываются в результате
- Разница между ними ~10-15 минут на 4-часовом маршруте
- Непонятно какой "правильный"

### 12. Влияние рельефа "+66%" — некорректная метрика

**Формула:**
```python
flat_time = total_distance / (60 / flat_pace)
elevation_impact = ((combined / flat_time) - 1) * 100
```

**Проблема:**
- `combined` уже включает fatigue
- Сравнивается "реальное время С fatigue" vs "идеальное время БЕЗ fatigue на плоском"
- Результат завышен

## Уточнение про Fatigue (Claude)

Проверил код — fatigue **опционально включается** через `apply_fatigue=True/False`:

```python
if apply_fatigue:
    self._fatigue_service = RunnerFatigueService.create_enabled(...)
else:
    self._fatigue_service = RunnerFatigueService.create_disabled()
```

Когда `fatigue=disabled`, `apply_to_segment` возвращает `multiplier=1.0`, так что fatigue действительно не добавляется.

**НО проблема в другом:** ВСЕ методы в totals накапливают время через цикл. То есть:
- `totals["strava_gap"]` = сумма strava_gap времён **только для RUN сегментов**
- `totals["tobler"]` = сумма tobler времён **только для HIKE сегментов**

Это не "полное время по методу", а частичное!

## Разбор персонализации

### Claude:

**Как работает GAP (формула):**
```
Для сегмента с градиентом +12%:
1. Берём flat_pace пользователя: 6:00/км
2. Смотрим в таблицу/формулу: gradient +12% → коэффициент 1.50
3. adjusted_pace = 6:00 × 1.50 = 9:00/км
4. time = distance / pace
```

Формула говорит: "На +12% ты будешь бежать в 1.5 раза медленнее чем на плоском".

**Как работает персонализация (lookup):**
```
Для сегмента с градиентом +12%:
1. Категория = "moderate_uphill" (8-15%)
2. Смотрим в профиль пользователя: avg_moderate_uphill_pace = 8:30/км
3. time = distance / 8:30
```

Lookup говорит: "На +12% ты реально бегаешь 8:30/км (по данным твоих Strava активностей)".

**Разница на практике:**

| Подход | Источник данных | Что учитывает |
|--------|-----------------|---------------|
| **GAP + flat_pace** | Формула × твой плоский темп | Усреднённое поведение 240k атлетов |
| **Lookup из профиля** | Твои реальные забеги на таких градиентах | Твою технику, твою силу, твои особенности |

**Пример:**
- GAP говорит: на +12% ты замедлишься в 1.5 раза → 9:00/км
- Твой профиль говорит: на +12% ты реально бегаешь 8:30/км

Ты быстрее "среднего атлета" на подъёмах! Lookup это учитывает, GAP — нет.

## Как это работает в hiking

### Claude:

**Tobler базовый:**
```
speed = 6 × exp(-3.5 × |gradient + 0.05|)  — формула, данные пользователя не нужны
```

**Tobler персонализированный:**
```
1. Считаем tobler_speed для градиента
2. Считаем ratio = user_flat_speed / 5.0 (стандартная скорость)
3. personalized_speed = tobler_speed × ratio
```

То есть персонализация в hiking **масштабирует формулу** по твоей скорости на плоском.

**Вариант 3.5**
персонализация (lookup) + Х — и к сожалению у нас есть тогда вариант три с половиной когда персонализация заполнена не полностью и нам не хватает каких-то данных по сегменту. И тогда, если есть персональные данные по сегменту, мы используем их, а если нет, нам нужно использовать какой-то из предыдущих вариантов.

## Подтверждение логики (Claude)

Проверил код — именно так работает!

Ключевой метод `_get_pace_extended()` в personalization_base.py:

```python
def _get_pace_extended(self, gradient_percent: float) -> float:
    category = self._classify_gradient_extended(gradient_percent)
    pace = self._get_pace_for_category(category)  # Lookup из профиля!

    if pace is not None:
        return pace  # ← Вариант 3: прямой lookup

    # Fallback
    return self._estimate_pace_for_gradient(gradient_percent)  # ← Вариант 3.5
```

`_estimate_pace_for_gradient()` в HikePersonalizationService:
```python
def _estimate_pace_for_gradient(self, gradient_percent: float) -> float:
    # Берём flat pace пользователя
    flat_pace = self.profile.avg_flat_pace_min_km
    flat_speed = 60 / flat_pace if flat_pace else DEFAULT_FLAT_SPEED_KMH

    # Считаем Tobler для этого градиента
    tobler_speed = tobler_hiking_speed(gradient_percent / 100)

    # Масштабируем по flat speed пользователя
    scale_factor = flat_speed / 5.0  # 5.0 = стандартный Tobler flat
    estimated_speed = tobler_speed * scale_factor

    return 60 / estimated_speed
```

### Вот точная картина:

| Вариант | Название | Логика | Когда используется |
|---------|----------|--------|-------------------|
| **1** | Базовый (формула) | Чистая формула, никаких данных пользователя | Нет профиля |
| **2** | Масштабированный | Формула × (user_flat / standard_flat) | Есть профиль, **НО** `use_extended_gradients=False` |
| **3** | Lookup | Прямой темп из профиля для категории | Есть профиль с данными для этой категории |
| **3.5** | Lookup + fallback | Lookup где есть, масштабированная формула где нет | Есть профиль, но не все категории заполнены |

## Итоговая структура

### 3 базовых метода Trail Running:

| Метод | Uphill | Downhill |
|-------|--------|----------|
| Strava GAP | Strava | Strava |
| Minetti GAP | Minetti | Minetti |
| Strava+Minetti | Minetti | Strava |

### 3 вариации:

1. **Персонализация** — lookup из профиля + fallback на масштабирование
2. **Переход на шаг** — после X% применяем hiking формулы (Tobler/Naismith)
3. **Усталость** — +% к времени после Y часов

### 4 варианта расчёта (для любого метода):

| Вариант | Когда используется | Логика |
|---------|-------------------|--------|
| **1** | Нет профиля | Чистая формула |
| **2** | Есть flat_pace, нет данных для категории | Формула × (user_flat / standard_flat) |
| **3.5** | Полный профиль | Lookup + fallback на Вариант 2 |
