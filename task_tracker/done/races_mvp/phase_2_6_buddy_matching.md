# Фаза 2.6: Buddy Matching — найди напарника для маршрута

## Контекст

Матчинг бегунов/хайкеров по совместимости на основе реальных GPS-данных (gradient-pace profile). Данные и алгоритмы — на стороне gpx_predictor. UI и клубная логика — на стороне ayda_run.

## Архитектура: кто что делает

### gpx_predictor (данные + расчёты)

**Уже есть:**
- `users.gradient_pace_profile` — JSONB, 11 бакетов темпа по градиентам
- `gap_calculator.py` — GAP расчёты
- Strava sync + профиль бегуна
- Activity patterns (дни, время, объёмы)

**Нужно создать:**

| Компонент | Описание | Сложность |
|-----------|----------|-----------|
| `compatibility_service.py` | cosine similarity между gradient-pace profiles | ~50 строк |
| Кластеризация | Группировка участников по темпу на конкретном маршруте (k-means или пороги) | ~100 строк |
| API endpoint | `POST /api/v1/matching/group` — список telegram_id + route_id → группы | ~50 строк |
| API endpoint | `GET /api/v1/matching/compatible?telegram_id=X` — совместимые юзеры (ступень 2) | ~80 строк |

**Общая оценка: 3/10 геморности.** Данные есть, алгоритм — cosine similarity + пороги.

### ayda_run (UI + клубная логика)

**Нужно создать:**

| Компонент | Описание | Сложность |
|-----------|----------|-----------|

**Общая оценка: 6/10 геморности.** Блокер — клубная инфраструктура (Фаза 6), которой ещё нет.

## Три ступени

### Ступень 1: Группировка внутри клуба

**Сценарий:** Тренер создаёт лонг → система предлагает группы по темпу.

```
Лонг в субботу: Кок-Жайляу
Участники: 12 человек

Рекомендованные группы по темпу на подъёмах:
🔴 Быстрая (4:30-5:30/км): @user1, @user3, @user7
🟠 Средняя (5:30-7:00/км): @user2, @user5, @user8, @user11
🟢 Комфортная (7:00+): @user4, @user6, @user9, @user10, @user12
```

**Алгоритм (упрощённый):**
1. Для маршрута лонга → определить ключевые градиенты (средний подъём)
2. Для каждого участника → достать его темп на этих градиентах из gradient-pace profile
3. Разбивка на 2-4 группы по порогам

**Зависимости:** Фаза 6 (Club Management) на ayda_run

### Ступень 2: Межклубный матчинг (после 50+ юзеров)

Юзер из одного клуба видит совместимых бегунов из другого клуба или без клуба. Opt-in обязателен.

**Алгоритм (полный):**
```
compatibility_score = w1 * pace_similarity + w2 * route_overlap + w3 * schedule_match + w4 * volume_ratio
```

**Зависимости:** Ступень 1 + 50+ юзеров из разных клубов

### Ступень 3: Публичный поиск (масштаб)

Полноценная фича на портале. Привязана к маршрутам и readiness flow.

**Зависимости:** Фаза 9 (портал) + Route Matching

## Privacy

- **Ступень 1:** Opt-in не нужен — Community Application exception покрывает клубный сценарий
- **Ступень 2-3:** Opt-in по умолчанию OFF
- Показываем: имя, грейд, "бегал этот маршрут N раз", совместимость по темпу
- НЕ показываем: GPS-треки, даты, пульс, Strava-профиль

## Порядок реализации

1. Ступень 1 — вместе с Фазой 6 (клубы) на ayda_run. gpx_predictor: ~200 строк (сервис + endpoint)
2. Ступень 2 — после валидации на клубах, 50+ юзеров
3. Ступень 3 — после портала

## Блокеры

- **Главный:** Фаза 6 (клубная инфраструктура) на ayda_run — без неё ступень 1 невозможна
- Для ступени 2-3: критическая масса юзеров
